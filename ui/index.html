Runs UTrack's 2-D scheme live in your browser: parcels ride the moisture-flux wind (viwve/tcw), rain out at the column rate (tp/tcw), and disperse via a Langevin velocity perturbation whose local amplitude is the ERA5 humidity-weighted vertical wind shear of the column (6 h decorrelation — the model's mixing timescale). This reproduces, statistically, the shear spreading of the full 3-D scheme. Winds are real ERA5 from … (looped daily). Emission rate scales linearly (passive tracer) — the pattern is what matters. For full 3-D Lagrangian physics, design a run and execute it on AWS.

Workflow

UTrack itself is a batch C model that needs hourly global ERA5 forcing — far too heavy for a browser. This workbench handles everything around a run:

  1. Design — paint a moisture source region, pick a scheme and dates, and export the release mask (.nc), the list.txt job line, and ready-to-paste run commands.
  2. Run on AWS — the repo's Docker image stages forcing and masks from S3, runs the model, and uploads results back to S3 (see README-DEPLOY.md).
  3. View — drop the output NetCDF here to explore where the released moisture precipitated.

Simulation schemes

Scheme numbers select the experiment configurations from Tuinenburg & Staal (HESS 2020). The two reference configurations recommended by the paper:

6Eulerian, 2-DVertically integrated budget variant
8 / 9 / 10LagrangianMixing-timescale sensitivity: 1 h / 24 h / 120 h

0–3LagrangianTrajectory-scheme sensitivity experiments (3 = isentropic levels)

Files and formats

  • Release mask — 721×1440 grid (0.25°, lat 90…−90, lon 0…359.75) with an integer release variable, 1 = source cell. Masks exported here are NetCDF-3 classic, which the model reads directly.
  • Output — the model writes released, allocated and insystem fields (mm of moisture per grid cell) as NetCDF-3; this viewer parses them fully client-side. Nothing you drop here leaves your browser.
  • Forcing — 5 ERA5 files per simulated day (~2.5 GB/day compressed); a run needs release days + 24 days. Download once to S3 with forcing/get_ERA5.py and reuse across runs.

Cost & sizing cheat-sheet

  • Memory: ~3–4 GB base + parcel arrays (shown live in the designer).
  • CPU: Lagrangian schemes are OpenMP-parallel — 8–16 vCPU instances work well.
  • A 5-day-release run ≈ 29 forcing days ≈ 73 GB of ERA5 on the instance or streamed from S3.

Technical summary

Three components. (1) The UTrack model — the published atmospheric moisture-tracking model (Tuinenburg & Staal, HESS 24, 2419–2435, 2020), a C/OpenMP batch code that tags moisture evaporating from user-defined source cells and follows it through ERA5 reanalysis weather until it precipitates; runs in Docker on AWS with S3 staging. (2) This workbench — a static app (AWS Amplify, no backend) that designs runs, and views model output entirely client-side: files you drop here never leave your browser. (3) The live simulator — a browser implementation of UTrack's 2-D scheme on preprocessed real ERA5 fields for instant emitter/run/reset experimentation; demonstrative, with the full 3-D model as the quantitative instrument.

Full model (batch, AWS)
  • Grid: global 0.25° (721 × 1440), 25 pressure levels (50–1000 hPa), hourly ERA5.
  • Forcing: 5 files per simulated day (~5.8 GB/day measured); a run needs release days + 24 consecutive days (≥ 25 days ≈ 145 GB).
  • Physics: parcels released ∝ ERA5 evaporation from masked cells; advected by 3-D winds (scheme-dependent); vertical mixing 1–120 h (reference schemes: 4 = Lagrangian 6 h, 7 = Eulerian 3-D); allocated to precipitation at rate P/PW. Moisture is a passive tracer — results are linear in the released amount.
  • Output: NetCDF-3 global fields released, allocated, insystem (mm per 0.25° cell, time-integrated).
  • Resources: ~3–4 GB RAM + parcel arrays; OpenMP-parallel; 8–16 vCPUs recommended.
  • CDS-format fixes in this port: unpacked floats (scale/offset defaults), viwve/viwvn names, surface-first pressure levels (auto-reversed), zip-wrapped single-level downloads (auto-merged).
  • Validation: run on real 2012-07-01 forcing — 4.41 mm released from the Utrecht cell, 67% allocated downwind (max in Siberia), mass budget closed. Browser-written masks verified against ncdump, python-netCDF4, and the compiled model.
Live simulator — methods & equations
  • Advection: ueff = viwve/tcw, veff = viwvn/tcw (moisture-flux-weighted column wind, m/s) — the model's own 2-D scheme.
  • Rainout: r = tp·1000/tcw per hour (capped 50%/h); each step a parcel deposits vol · r · Δt/3600.
  • Dispersion: per-parcel Langevin (Ornstein–Uhlenbeck) velocity perturbation u′ ← u′(1 − Δt/τ) + σ·√(2Δt/τ)·ξ with τ = 6 h (the model's reference mixing timescale) and σ(x,y,t) = ERA5 humidity-weighted std-dev of wind across the column's 25 levels (mean ≈ 3.2 m/s, p95 ≈ 8 m/s). This reproduces statistically the vertical-shear spreading of the 3-D scheme. The UI's 0.5×/1×/1.5× selector is a sensitivity test.
  • Time step: Δt = 300 s; playback 1/4/12 sim-hours per wall-second; parcel cap 30,000.
  • Budget: emitted = airborne + rained out (acre-feet; 1 ac-ft = 1233.48 m³) — closure is exact by construction and displayed live.

Source: ERA5, the 15th of each month of 2025, looped daily (~4.8 MB per month, fetched on demand). Rainout and the deposition map stay at 0.5° so ERA5's real orographic precipitation — windward enhancement and rain shadows — survives downsampling: mountains enter through the data, not a parameterization. The basemap hillshade is ERA5 orography and is purely visual.

Known limitations (state when presenting)
  • One representative day per month, looped — long runs re-experience that day's weather rather than an evolving synoptic sequence.
  • 2-D column transport — vertical structure is statistical (shear dispersion), not resolved.
  • 0.5°/1° resolution — major ranges and shadows resolve; individual valleys don't. Quantitative claims belong to full 3-D runs at 0.25° on AWS.
  • Emission is a passive-tracer scaling; it does not feed back on the weather (valid for amounts small relative to atmospheric moisture).
Architecture & deployment
  • Hosting: AWS Amplify, auto-deploy from GitHub main; static files only, no backend, no accounts. Dropped result files are parsed in-browser and never uploaded.
  • Stack: vanilla JS + Canvas; in-house NetCDF-3 reader/writer (ui/netcdf.js); validated accessible palette; light/dark themes.
  • Model runs: Docker (Ubuntu 24.04 + libnetcdf) on EC2 or AWS Batch; entrypoint syncs forcing/masks from S3 and uploads results (README-DEPLOY.md).
  • Credentials: CDS token via env vars from Secrets Manager; AWS via IAM roles; nothing in the repo or image.
  • Pipeline scripts: forcing/get_ERA5.py (model forcing), forcing/make_sim_assets.py (simulator months), forcing/make_terrain.py (hillshade).
